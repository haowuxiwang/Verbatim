## 3. 本轮新发现

以下是本轮深度审查中仍然值得记录的风险点，按严重度排序。

### 3.1 中风险：OCR token 会在 DPAPI 不可用时静默降级为明文落盘

证据：

- `core/ocr_client.py:238`
- `core/ocr_client.py:239`
- `core/ocr_client.py:257`
- `tests/test_ocr_client.py:150`

现象：

- 当前实现会先尝试 DPAPI 加密
- 若加密结果无法稳定解密回原文，就回退到 `token_storage = "plain"`
- 该降级对功能稳定性是合理的，但对安全性而言，意味着敏感 token 会落入本地明文配置文件

影响：

- 功能上更稳，不会因为系统 DPAPI 状态异常而直接导致配置不可用
- 但如果用户默认认为“Windows 下一定加密存储”，这个假设已经不成立
- 对共享机器、受控办公环境、桌面取证场景存在额外暴露面

建议：

- 至少在 UI 设置页或首次保存时明确提示当前 token_storage 模式
- 若检测到回退为 plain，建议加红色安全提示而不是静默接受
- 更长期可考虑引入受控的加密不可用策略，例如“拒绝保存”与“允许明文保存”交由用户显式选择

### 3.2 中风险：release gate 已明显增强，但仍然偏向源码态烟测，不足以代表完整交付质量

证据：

- `scripts/pyinstaller_release_gate.py:9`
- `scripts/pyinstaller_release_gate.py:47`
- `scripts/pyinstaller_release_gate.py:64`
- `scripts/pyinstaller_release_gate.py:105`

现象：

- source smoke tests 目前只覆盖 3 个测试文件
- exe smoke test 仍然只是“是否能拉起并存活若干秒”
- 当前常用验收命令仍是 `--skip-exe-run`

影响：

- 现在的 gate 已经比 2026-03-25 强很多，但仍然主要在验证“源码态没坏”
- 对打包产物上的 OCR worker、真实资源路径、冻结态 UI 行为、长时间稳定性，覆盖仍然不足
- 如果未来把“gate_pass=true”误解成“发布候选已充分验证”，容易再次形成错误安全感

建议：

- 为 exe 态补一组真正的业务 smoke path，而不仅是启动存活检测
- 把 OCR on/off、compare history replay、超时恢复这几条关键路径至少覆盖一条打包态冒烟
- 将 `--skip-exe-run` 明确标为“开发态快速检查”，不要让它成为默认发布依据

现象：

- 当前对“扫描版 PDF / 无文本层”的处理已经能打印 warning 并触发 OCR 倾向
- 但代码中仍保留 `TODO`
- 同一类问题在 `_load_page_data()`、`_update_left_page()`、`_update_right_page()` 中分散出现

影响：

- 目前功能上可用，但状态来源不统一
- 后续如果继续叠加 OCR 推荐、质量评分、手动复核门槛，很容易在多个入口出现重复判断或分叉行为
- UI 层目前更多依赖即时输出，而不是一个稳定的页面质量状态对象

建议：

- 把“页面文本层质量 / 是否扫描版 / 是否建议 OCR / 是否要求人工复核”统一沉到 view-model 或 service 层
- UI 只消费状态，不在多个入口自行推断
- 这会让后续 warning 展示、按钮态、compare gate 逻辑更加一致
