# 20260311 代码审查

## Findings (按严重程度排序)

1. **High**：OCR 超时后后台线程不会被回收，可能累积并继续占用 CPU/内存，导致持续卡顿。  
   证据：`_run_in_background_with_ui_pump` 在超时时直接抛异常，但没有任何取消或回收机制，线程仍会继续运行；`_try_ocr_text` 会在多个 variant/engine 中重复调用该函数。  
   位置：[main_window.py](/D:/learn/codex/Verbatim_dev/app/main_window.py#L3285)  
   建议：为后台任务加入取消/超时中断机制（例如 `QThread` + `requestInterruption`），或把长任务集中到单一工作线程并复用，避免超时后继续跑。

2. **Medium**：OCR 结果缓存对 bbox 进行整数取整，轻微拖动（<1px）会命中旧缓存，导致“不同选区得到同一 OCR 文本/坐标”，产生错误差异与高亮。  
   位置：[main_window.py](/D:/learn/codex/Verbatim_dev/app/main_window.py#L3261)  
   建议：改为按更精细的四舍五入（例如 0.1px）或加入 bbox 尺寸/面积变化阈值；或者在缓存 key 中加入 text-layer 片段 hash 作为二次校验。

3. **Medium**：OCR 行框被等分为字符 bbox，且按 `len(text)` 直接均分宽度（包含空格/标点），会造成高亮扩散或错位，尤其是混合中英文/空格的行。  
   位置：[main_window.py](/D:/learn/codex/Verbatim_dev/app/main_window.py#L3099)  
   影响：OCR 标注精度下降，用户感知“整行不准”。  
   建议：过滤空白字符不分配 bbox，或按字符类别调整宽度权重；可考虑使用 OCR 引擎输出的更细粒度坐标（如字级或词级）。

4. **Low**：`_run_in_background_with_ui_pump` 在 OCR 过程中持续 `processEvents()`，可能允许用户改变选区/页码，导致一次 compare 过程中输入状态被改变，造成结果不稳定。  
   位置：[main_window.py](/D:/learn/codex/Verbatim_dev/app/main_window.py#L3285)  
   建议：compare 期间禁用选区交互，或在 compare 开始时冻结选区并在结束时恢复。

## 其他观察
- OCR 与纯文本链路逻辑已明显增强，但可见性与稳定性仍强依赖 OCR 质量与缓存一致性策略。
