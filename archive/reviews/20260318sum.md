# 20260318 Summary - OCR Spans Render Reliability

Date: 2026-03-18

## Context (What happened)
- During GUI tests, the machine froze. Suspected root causes included:
  - OCR subprocess not released
  - GUI repeatedly creating resources without freeing
  - Debug span drawing causing UI overload
  - OCR image too large leading to memory blow
- Critical symptom: logs reported spans, but UI showed no boxes. No UI提示, making the visualization untrustworthy.
- Requirement: add UI-side log `[SPANS_RENDER] count=... scale=... canvas=(w,h)` to verify whether UI consumes spans, and identify the top 3 likely bugs when spans exist but are invisible: coordinate mismatch, bbox format misunderstanding, UI locked/overwritten.

## Current Progress (Implemented)
### 1) UI spans render logging
- Added UI-side `[SPANS_RENDER]` logging via `log_event` with trace_id/side/count/scale/canvas.
- Emission is change-triggered (only when count/scale/canvas changes) to avoid log spam.

### 2) Render spans independent of interaction state
- Previous behavior: spans rendering was coupled to selection/compare state. If compare overlays were cleared or coordinates marked unreliable, spans were also cleared, causing “spans exist but no boxes”.
- Now: spans rendering is decoupled from selection/compare state in debug mode:
  - If there are spans but no diff ops, UI still renders spans-only overlays.
  - Debug spans overlays are merged after “unlocatable” logic so they’re not wiped by compare state.

### 3) UI warning when spans exist but not locatable
- When spans exist but coordinates fail validation or are unreliable, a warning is added to UI warnings list.
- This prevents “no boxes = no data” misinterpretation.

### 4) Spans lifecycle decoupled from selection
- Introduced OCR spans metadata cache:
  - Keyed by OCR result cache key (pdf/page/source_bbox/side)
  - Stores `pdf`, `page`, `clip_bbox`, `source_bbox` used to generate spans
- Rendering eligibility now depends on OCR result metadata instead of current selection:
  - If selection is `None`, spans can still render (data-first view)
  - If selection exists but does not match source_bbox, spans are not rendered (prevents misleading overlay)
  - If page/pdf mismatches, spans are not rendered

## Current Situation (Behavior now)
- If OCR spans exist, UI will render spans-only overlays even when selection is cleared or compare overlays are reset.
- If spans are “unlocatable” (validation failed or coords unreliable), UI adds a warning so user understands the data exists but cannot be trusted for positioning.
- Logs contain `[SPANS_RENDER]` confirming UI consumption of spans.

## Potential Issues & Risk Points (Detailed)
1) **Coordinate System Mismatch (Highest risk)**
   - OCR spans may be in image-pixel coordinates, clip-local coordinates, or normalized space.
   - We assume spans are in clip image coordinates and convert via `clip_bbox + span/zoom`.
   - If OCR engine outputs are in different coordinate space, overlays will be offset or off-canvas.
   - Risk: spans appear missing even though data exists.

2) **BBox Format Misinterpretation**
   - OCR output may be `(x, y, w, h)` or polygon points; `_normalize_box` may interpret incorrectly.
   - This can produce invalid spans (x1<=x0, y1<=y0) or misplaced spans.
   - Validation will mark as invalid, and UI will show “unlocatable” warning.

3) **UI Overlay Clearing / State Overwrite**
   - There are multiple places that call `set_overlays([])` in selection/page-change flows.
   - Although spans-only render is now independent, some flows may still clear overlays before spans get re-rendered.
   - Risk: short flicker or temporary disappearance if spans rendering is not re-triggered.

4) **Stale Spans Across Page/Selection**
   - We now guard against rendering spans if pdf/page mismatch or bbox mismatch.
   - However, if OCR cache keys collide due to rounding, spans could be incorrectly considered relevant.
   - Risk: wrong spans shown for a different region.

5) **Memory Pressure**
   - OCR spans cache and meta cache can grow; repeated OCR over many regions may accumulate memory.
   - Not yet adding explicit eviction beyond `_clear_cached_diff_result` and page switches.
   - Risk: long session could grow memory footprint.

6) **UI Freeze still possible**
   - If OCR image renders are too large or OCR subprocess stalls, UI can still freeze.
   - We added logging and diagnostics but not full resource caps or subprocess cleanup.

## Tool Architecture & Construction Rationale
### High-level approach
- This tool is built as a **dual-PDF comparison UI** with OCR fallback.
- The core pipeline:
  1. User selects left/right PDF regions (selection bboxes)
  2. Extract text from PDF text layer
  3. If quality is low, OCR fallback is triggered
  4. OCR output is normalized and compared
  5. Differences are visualized as overlays and list items

### Why this architecture?
- **Hybrid text + OCR**: text-layer extraction is fast and accurate for digital PDFs, OCR is necessary for scans.
- **Region-based selection**: reduces OCR cost and improves precision, avoids full-page OCR.
- **Quality gates and warnings**: prevents over-trusting OCR in low-confidence cases.
- **Diff overlays**: aligns differences to document layout for fast review.

### Is this approach optimal?
- It is **pragmatically optimal** for accuracy/performance tradeoff in document comparison:
  - OCR only when needed
  - UI overlays for fast review
  - Quality warnings to prevent false certainty
- But it is **not theoretically optimal** in all scenarios:
  - OCR coordinate systems can vary; robust alignment needs explicit calibration or model-specific adapters.
  - A dedicated layout-aware pipeline (e.g., structured OCR with explicit coordinate standardization) would reduce mismatch risk but costs more engineering and runtime.
  - For extreme large PDFs or repeated comparisons, a caching and eviction policy would be needed for memory stability.

### Why we changed spans rendering strategy
- Original design bound rendering to interaction state (selection/compare), which caused spans visibility to be unreliable.
- For debugging and correctness, **data state must dominate over interaction state**.
- Hence spans rendering is now driven by OCR result metadata, not current selection state.
- This ensures: if OCR produced spans, UI reflects them unless they are unsafe to show.

## Open Questions / Next Steps
- Should we add explicit coordinate-system flags per OCR engine to normalize spans precisely?
- Should we implement spans cache eviction (LRU or per-page)?
- Should we add UI-level “spans not locatable” overlay badge on the canvas itself, not just in diff list?
