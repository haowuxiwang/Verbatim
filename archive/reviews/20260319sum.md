# Verbatim 仓库审查总结（2026-03-19）

## 一、结论摘要

当前版本在功能链路上已达到“可用”：

- PDF 加载与框选：可用
- OCR 决策链路：可用
- spans 生成：可用
- diff 输出：可用
- UI 渲染：基本可用

但当前不具备发布条件。核心阻断点不是“有没有结果”，而是“结果是否稳定、是否可解释、是否一致可复现”。

综合判定：

- 功能完整性：通过（可进入稳定性整治阶段）
- 可观测性：部分通过（关键生命周期日志缺失）
- 稳定性：不通过（P0）
- 打包一致性：不通过（P0）
- UI 可用性：不通过（P0/P1 混合）

---

## 二、P0 级阻断问题（必须先解决）

### P0-1：源码运行与 exe 运行存在默认行为反转

在本地 OCR 路径上，是否使用子进程隔离的默认值取决于 `sys.frozen`：

- 非冻结（源码运行）默认隔离开启
- 冻结（exe 运行）默认隔离关闭

这会直接导致“命令行测试稳定，但打包后更容易卡死/假死”的现象。

风险：

- 线程内 OCR 推理一旦阻塞，UI 主线程体验显著恶化
- 用户感知为“同版本不同行为”

### P0-2：后台超时不等于后台任务被回收

当前后台执行超时后会抛出异常返回 UI，但原后台线程并不具备可中断回收机制。连续触发后，后台可能累计运行负载。

风险：

- 连续比对后 CPU/内存压力持续上升
- UI 响应越来越慢，最终接近“卡死”

### P0-3：UI 线程上仍有重操作

页面渲染、页面解析、历史持久化等关键路径仍在主线程执行，OCR 路径又存在 `processEvents` 驱动的长等待。

风险：

- 用户操作与后台状态交叉，出现状态漂移
- 在高负载机器上易出现短时无响应或长卡顿

### P0-4：打包门禁不足以兜底稳定性

当前发布门禁偏“能启动就算过”，未覆盖真实路径：

- 未强制执行连续 compare（含 OCR on/off）
- 未做资源峰值阈值门禁（CPU/内存/句柄）
- 未做“超时后恢复能力”强校验

结果：

- 可执行文件能启动，但不代表可稳定交付

## 三、UI 专项问题（基于你提供截图 + 代码链路分析）

你反馈的三类现象是准确的：

1) 差异看不清  
2) 噪音多  
3) OCR 后 UI 无法返回“增删改减”标记

以下是具体分析。

### 3.1 差异“看不清”的原因

#### 3.1.1 视觉层级冲突，选区高亮掩盖差异语义

UI 同时展示：

- 选区框（橙/蓝/绿）
- 差异块（低透明填充）
- 徽标（增删改格）

截图中可见大面积选区边框与背景色非常显眼，而差异块是低 alpha 的轻填充，用户第一眼看到的是“框选范围”，不是“差异点”。

#### 3.1.2 差异填充透明度过低 + 徽标过小

当前绘制策略是“轻背景 + 左边细条 + 16px 徽标”，在高 DPI 或复杂 PDF 背景上辨识度不足，用户需要放大很多才能看清。

#### 3.1.3 信息噪声优先级高于差异条目

差异列表顶部会先插入质量警告、置信度、分隔线等信息。它们视觉上很抢眼，且经常占据列表前几项，用户第一屏看到的是“警告块”，不是“可定位差异”。

---

### 3.2 “噪音多”的原因

#### 3.2.1 OCR spans 转字符框使用等宽切分，天然会引入定位噪音

当前 spans 到字符 bbox 的映射采用“按文本长度均分行框宽度”。这在中英混排、空格、标点、比例字体场景下会出现扩散与错位，视觉上像“整行都差异”。

#### 3.2.2 OCR 缓存 key 的 bbox 取整过粗

bbox 缓存采用整数像素四舍五入，轻微拖动可能命中旧缓存，导致“看似新选区，实则旧结果”。

表现为：

- 文本与高亮不一致
- 噪音难以复现但频繁出现

#### 3.2.3 低可信场景的警告与结果混在同一列表

当 OCR 低可信时，会加入大量状态告警与降级说明。若没有单独的“告警区/诊断区”，用户会把这些与真实差异混为一谈，主观感受即“噪音过多”。

---

### 3.3 OCR 后“增删改减标记回不来”的原因

这是当前最关键的 UI 语义问题之一。

#### 3.3.1 设计上主动清空了不可置信坐标的 overlay

一旦判定 OCR 坐标不可靠（或 OCR 应用但无可定位坐标），系统会清空对应侧 overlays，只保留“不可定位”提示徽标。

这会直接导致用户感知：

- 有 diff 结果
- 但画布上看不到增删改定位
- 只看到状态标签（例如 OCR/无）

#### 3.3.2 列表首项自动选中可能落在“非差异项”

差异列表构建后默认选第 0 行，但第 0 行常常是“置信度/警告头/分隔线”（不可定位项，不带 diff 数据）。

结果：

- 用户以为系统已定位
- 实际未触发焦点定位逻辑
- 进一步强化“标记没回来”的感受

## 四、你截图中可直接观察到的 UI 问题映射

### 4.1 右侧面板信息密度过高

- 比对历史、差异列表、详情区域纵向堆叠紧密
- 文本字号小、可视区域窄、条目截断明显
- 重要信息（可点击差异）与状态信息（告警）混杂

### 4.2 差异语义不够“第一眼可读”

- 画布上大框明显，小差异不明显
- 列表中的“增删改”虽然有文案，但画布定位反馈弱
- 用户在 OCR 路径下难以建立“列表条目 <-> 文档位置”的稳定映射

### 4.3 OCR 场景下的可解释性断裂

- 系统实际上做了“不可定位保护”
- 但用户看到的是“没有标记”
- 缺少更强显式提示（例如“本轮仅文本差异可用，几何定位已降级”）

## 五、可观测性现状与缺口

### 已有进展

- OCR 决策、候选评分、失败分类、状态机日志已补齐较多
- spans render 有调试日志
- compare 起止事件有日志

### 仍缺失的关键点

- render 生命周期完整链路日志（开始/成功/耗时/失败）
- UI 卡死前后状态快照（trace_id、页码、bbox、当前 mode、OCR route、缓存大小）
- 主线程繁忙采样（例如心跳/事件循环延迟）

结论：目前能看“发生了什么”，但仍难以精确回答“为什么在那一刻卡死”。

---

## 六、打包与命令行不一致问题总结

### 主要根因

- 构建配置入口多且不统一（多个 spec）
- 不同 spec 对 datas（core/umi/ocr_runtime）打包策略不同
- 冻结态默认参数与源码态不一致（尤其 OCR 隔离）
- 发布门禁未覆盖真实业务路径

### 直接后果

- 测试结论不可直接迁移到 exe
- 用户现场复现与研发环境复现出现明显偏差

---

## 七、风险分级

### P0（发布阻断）

1. 冻结态与源码态 OCR 隔离默认值不一致  
2. 后台超时后线程回收不可证  
3. UI 线程重操作导致卡顿/假死风险  
4. OCR 后“不可定位时无增删改标记”的主流程可解释性不足  
5. 打包门禁无法覆盖真实连续比对压力路径

### P1（需尽快治理）

1. 差异视觉层级不合理（看不清）  
2. 列表信息噪声过高（可读性差）  
3. OCR spans 到字符框映射精度不足导致噪音  
4. 缓存粒度导致轻微拖动命中旧结果  
5. 右侧面板布局在高密度场景下可用性差

---

## 八、阶段结论（给当前决策用）

当前版本可以继续内部验证，但不建议对外发布。

建议决策：

- 先按 P0 关闭稳定性与一致性风险
- 再做 UI 可读性与可解释性改造（重点是 OCR 降级场景）
- 最后才进入新一轮打包验收

一句话总结：

功能“能跑通”，但在高负载与 OCR 降级场景下，系统还没有达到“可稳定、可解释、可发布”的工程标准。
