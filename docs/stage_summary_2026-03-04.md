# Verbatim 阶段压缩总结（截至 2026-03-04 16:11）

## 1. 当前方向与策略结论
- 当前主策略已切换为：`云端 OCR 单引擎（Paddle API） + 低可信降级 REVIEW + 历史可追溯`。
- 本地 OCR 已移除（含 `ocr/` 目录），避免本地低质量结果带来的噪音和打包臃肿。
- 在你给出的真实日志场景里，噪音核心触发点主要是云端超时链路和低可信文本误入差异引擎；本阶段已做强约束（超时降级、不硬报差异）。

## 2. 本阶段已完成工作（代码与行为）

### OCR 与比对策略
- 云端优先改为云端唯一主路径：比对流程不再执行本地 Tesseract 分支。
- OCR 超时/失败时，优先输出 `REVIEW`（需人工复核），不再强行给硬差异结论。
- OCR 失败时不再进行“扩框重试导致大面积噪音”链路（避免把无关文本卷入 OCR）。
- OCR/低质量场景默认关闭字段映射（减少误报放大）。

### 稳定性与线程
- 背景任务超时链路已收敛，避免 `QThread destroyed while running` 类崩溃路径。
- OCR预算和单次超时都有上限，避免用户长时间无反馈卡顿。

### 历史与回放
- 已实现比对历史持久化：`mappings/compare_history.json`。
- 已接入 GUI 的“比对历史”列表与回放入口。
- 历史记录字段扩展为可回放所需：状态、摘要、差异数量、OCR参与、页码、选区、告警数、`diff_ops`、左右区域文本、OCR应用标记。

### 可观测性
- 日志中能区分：OCR 路由、超时、质量告警、REVIEW 降级。
- 当前已能从日志判断“是算法问题还是链路问题（如代理/网络）”。

## 3. 当前仍存在的问题（与你反馈一致）
- UI 给人的稳定感仍不足：快速切换历史项时，视觉表现还有“跳动/错位感”风险。
- 真实场景仍可能出现噪音偏高（尤其云端超时后的低质量区域）。
- 差异结果虽然有 REVIEW 保护，但用户对“为何是 REVIEW”的可解释性仍需强化。

## 4. 对标工业级软件的差距

### A. 结果质量门禁（未完全工业化）
- 缺少固定样本集下的日常门禁指标：误报率、漏报率、弃权率、稳定性波动。
- 目前更多是“人工日志判断”，尚未形成持续质量看板。

### B. 交互可靠性（部分达成）
- 已有历史持久化，但“历史回放体验”还需进一步打磨（更明确状态、重放一致性、异常提示）。

### C. 云端链路鲁棒性（关键短板）
- 当前结果高度依赖网络质量；代理/出口不稳定会显著影响可用性。
- 需要更系统的网络侧策略（重试策略、分级超时、链路健康提示）。

### D. 架构治理（中等）
- 代码可用性提升明显，但 `app/main_window.py` 仍然较大（可维护性风险仍在）。

## 5. 当前 TodoList 完成度（阶段视角）

### 已完成
